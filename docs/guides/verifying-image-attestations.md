# Verifying and reading image SBOM and provenance

This guide shows, from a user's perspective, how to read and verify the
**SBOM** and **build provenance** attached to the CSSC Dashboard application
images, using **both** ways those attestations are published:

1. **Embedded** in the image index as BuildKit *attestation-manifests* — read
   with `docker buildx imagetools inspect`.
2. As **OCI 1.1 Referrers-API artifacts** attached with `oras attach` — discover
   with `oras discover` and pull with `oras`.

It is the task-oriented companion to the reference document
[image attestations](../reference/image-attestations.md), which describes what
the `build / cssc-dashboard` workflow produces. If you only want the summary,
read that document; if you want to *inspect* or *verify* the attestations, read
this one.

## Background: two representations

The build workflow produces, **per platform** (`linux/amd64`, `linux/arm64`):

| Attestation | Predicate type | Generated by |
| --- | --- | --- |
| SPDX SBOM | `https://spdx.dev/Document` | `docker buildx build --sbom=true` |
| SLSA provenance | `https://slsa.dev/provenance/v0.2` | `docker buildx build --provenance=mode=max` |

Each attestation is stored **twice**:

- **Embedded** — BuildKit writes it as an *attestation-manifest* inside the
  multi-arch image index (an entry with `platform: unknown/unknown`, linked to
  the platform image by the `vnd.docker.reference.digest` annotation). This is
  what `docker buildx imagetools inspect` reads.
- **Referrer** — the workflow extracts the same in-toto statement and re-attaches
  it to the *platform manifest* with `oras attach`
  (artifact type `application/vnd.in-toto+json`, annotation
  `in-toto.io/predicate-type=<predicate>`). This is what `oras discover` and the
  registry `/v2/.../referrers/<digest>` endpoint return.

Both point at the *same* in-toto statements; you can use whichever your tooling
supports. Referrers attach to the **platform** manifest, not the tag/index, so
most commands below need a per-platform digest first.

Throughout this guide the example image is:

```text
ghcr.io/toddysm/apps/cssc-dashboard/packages-service:0.1
```

Substitute `issues-service` or `dashboard-web` for the other services.

## Prerequisites

Install and authenticate the following CLIs:

- [`docker`](https://docs.docker.com/get-docker/) with Buildx (bundled with
  modern Docker) — reads the embedded attestations.
- [`oras`](https://oras.land) v1.2+ — discovers and pulls referrers.
- [`crane`](https://github.com/google/go-containerregistry/tree/main/cmd/crane)
  — lists platform digests (any manifest tool works; `docker buildx imagetools
  inspect --raw` is an alternative).
- [`jq`](https://jqlang.github.io/jq/) — parses JSON.

The CSSC Dashboard images are public, so anonymous reads work. If you are
reading from a private repository, log in first so both `docker` and `oras`
pick up the credentials:

```bash
echo "$GITHUB_TOKEN" | docker login ghcr.io -u "$USER" --password-stdin
```

## Step 1 — Resolve the per-platform digests

List the real platform manifests in the index (skip the `unknown/unknown`
attestation-manifests):

```bash
IMAGE=ghcr.io/toddysm/apps/cssc-dashboard/packages-service:0.1

crane manifest "$IMAGE" \
  | jq -r '.manifests[]
      | select(.platform.os != "unknown")
      | "\(.platform.os)/\(.platform.architecture)\t\(.digest)"'
```

Example output:

```text
linux/amd64	sha256:aaaa...
linux/arm64	sha256:bbbb...
```

Pick the digest for the platform you care about and keep it handy:

```bash
REPO=ghcr.io/toddysm/apps/cssc-dashboard/packages-service
PLATFORM_DIGEST=sha256:aaaa...   # from the list above
```

## Step 2 — Read the embedded attestations

`docker buildx imagetools inspect` resolves the embedded attestation-manifests
for you and returns the in-toto statements keyed by platform.

Read the **SBOM**:

```bash
docker buildx imagetools inspect "$IMAGE" --format '{{ json .SBOM }}' | jq .
```

Read the **provenance**:

```bash
docker buildx imagetools inspect "$IMAGE" --format '{{ json .Provenance }}' | jq .
```

Each entry is an in-toto statement whose `.predicate` holds the actual document
(the SPDX BOM, or the SLSA provenance). For example, to list the packages in the
SBOM for one platform:

```bash
docker buildx imagetools inspect "$IMAGE" --format '{{ json .SBOM }}' \
  | jq -r 'to_entries[]
      | .key as $plat
      | (.value.SPDX.packages // .value.packages // [])[]
      | "\($plat)\t\(.name)\t\(.versionInfo // "")"' \
  | head
```

> The exact shape returned by `--format '{{ json .SBOM }}'` can vary by Buildx
> version (it may be keyed by platform, or wrap the document under `.SPDX`). Pipe
> to `jq .` first to see the structure, then adjust the filter.

## Step 3 — Read the referrer attestations

### 3a. Discover the referrers

List everything attached to the platform manifest:

```bash
oras discover --format tree "$REPO@$PLATFORM_DIGEST"
```

You should see two `application/vnd.in-toto+json` referrers — one for the SBOM
and one for the provenance.

### 3b. Find the referrer digest for a predicate type

Select the referrer whose predicate type you want. This filter is robust across
`oras` output shapes:

```bash
# SBOM referrer:
SBOM_REF=$(oras discover --format json "$REPO@$PLATFORM_DIGEST" \
  | jq -r '[.. | objects
      | select(.artifactType? == "application/vnd.in-toto+json")
      | select(.annotations["in-toto.io/predicate-type"] == "https://spdx.dev/Document")]
      | .[0].digest // empty')

# Provenance referrer:
PROV_REF=$(oras discover --format json "$REPO@$PLATFORM_DIGEST" \
  | jq -r '[.. | objects
      | select(.artifactType? == "application/vnd.in-toto+json")
      | select(.annotations["in-toto.io/predicate-type"] == "https://slsa.dev/provenance/v0.2")]
      | .[0].digest // empty')

echo "SBOM referrer:       $SBOM_REF"
echo "Provenance referrer: $PROV_REF"
```

### 3c. Pull the in-toto statement

The simplest way is `oras pull`, which writes the statement to a file named by
its title annotation (`sbom.spdx.json` / `provenance.slsa.json`):

```bash
oras pull -o ./attestations "$REPO@$SBOM_REF"
oras pull -o ./attestations "$REPO@$PROV_REF"
ls ./attestations
```

If you prefer to stream the raw statement without writing intermediate files,
fetch the referrer manifest's first layer directly:

```bash
LAYER=$(oras manifest fetch "$REPO@$SBOM_REF" | jq -r '.layers[0].digest')
oras blob fetch --output - "$REPO@$LAYER" | jq .
```

### 3d. Inspect the statement

Every pulled file is an [in-toto Statement](https://github.com/in-toto/attestation):

```bash
# What kind of statement is it, and what does it describe?
jq '{predicateType, subject}' ./attestations/sbom.spdx.json

# The SPDX document itself:
jq '.predicate' ./attestations/sbom.spdx.json | jq '.name, .packages | length'

# Provenance: builder, source, and materials (base image, etc.):
jq '.predicate.builder, .predicate.invocation.configSource, .predicate.materials' \
  ./attestations/provenance.slsa.json
```

## Step 4 — Sanity-check what you read

Even without cryptographic signatures you can confirm the attestations are
internally consistent:

1. **The referrer describes the right image.** The referrer's `subject` digest
   must equal the platform digest you queried:

   ```bash
   oras manifest fetch "$REPO@$SBOM_REF" | jq -r '.subject.digest'
   # -> must equal $PLATFORM_DIGEST
   ```

2. **The embedded and referrer statements agree.** They are extracted from the
   same build, so the SBOM package set and the provenance `subject` digest should
   match between Step 2 and Step 3.

3. **Provenance points at the expected base image.** The SLSA `materials` (or
   `resolvedDependencies`) should include the golden base image digest, which is
   also recorded in the image's `org.opencontainers.image.base.digest`
   annotation:

   ```bash
   crane manifest "$REPO@$PLATFORM_DIGEST" \
     | jq -r '.annotations["org.opencontainers.image.base.digest"]'
   ```

4. **Revision matches the commit.** `org.opencontainers.image.revision` and the
   provenance source revision should be the Git SHA the image was built from.

## Copying attestations with the image

The embedded attestation-manifests move automatically when you copy the whole
index. The **referrers**, however, are separate manifests, so use a
referrers-aware copy to bring them along:

```bash
oras cp -r "$REPO:0.1" "another-registry.example.com/mirror/packages-service:0.1"
```

This is the same mechanism the repository's mirror/promote workflows use to carry
scan-report referrers between `quarantine/*` and `golden/*`.

## A note on trust and signatures

These attestations are **not cryptographically signed**. Reading them proves what
the build recorded and lets you cross-check consistency, but it does **not** prove
the image came from this repository the way a signature would. Treat the steps
above as *inspection and integrity checks*, not signature verification.

Signing the attestations (for example with keyless
[cosign](https://docs.sigstore.dev/) and GitHub OIDC, verified via
`cosign verify-attestation`) is a possible future enhancement and is out of scope
for the current build.

## Troubleshooting

- **`oras discover` returns nothing.** Referrers attach to the *platform*
  manifest, not the tag. Make sure you passed `$REPO@$PLATFORM_DIGEST`, not
  `$REPO:0.1`. Also confirm `oras version` is 1.2 or newer.
- **`docker buildx imagetools inspect ... .SBOM` is empty.** The image predates
  the SBOM/provenance work, or you inspected a single-arch reference. Inspect the
  multi-arch tag, not a `@sha256:` platform digest.
- **`unauthorized` / `denied`.** Log in to GHCR (see [Prerequisites](#prerequisites)).
  For private repositories, both `docker` and `oras` read `~/.docker/config.json`.
- **The pulled file is a DSSE envelope, not a bare statement.** Some producers
  wrap the in-toto statement in a DSSE envelope (`.payload` is base64). Decode it
  with `jq -r '.payload' file | base64 -d | jq .`. The CSSC Dashboard referrers
  are bare statements, so this is only relevant for images from other tools.
